# Staging Module — Invariants

| ID | Rule | Enforcement |
|---|---|---|
| INV-STAGING-001 | Le contenu pivot est stocké en forme canonique (ADR-0007), montants `decimal` préservés, AUCUN float/double ; le round-trip write→read restitue fidèlement le JSON canonique | `FileSystemPayloadStagingStore.WriteAsync`/`ReadAsync` (le JSON canonique est stocké/relu tel quel) ; `CanonicalJson` (échelle decimal préservée) — testé `FileSystemPayloadStagingStoreTests` |
| INV-STAGING-002 | Le payload est CHIFFRÉ au repos, avec une clé dérivée par tenant (isolation cryptographique inter-tenants — CLAUDE.md n°9/10) ; aucun contenu fiscal en clair sur le disque | `FileSystemPayloadStagingStore` (`IDataProtectionProvider.CreateProtector(purpose, tenantId)`, `Protect`/`Unprotect`) — testé (octets sur disque ≠ JSON clair) |
| INV-STAGING-003 | À la lecture, le `payload_hash` est RE-VÉRIFIÉ : un contenu altéré (empreinte recalculée ≠ attendue) OU un blob chiffré illisible (AEAD) est REJETÉ (`StagedPayloadIntegrityException`), jamais servi silencieusement | `FileSystemPayloadStagingStore.ReadAsync` (recalcul `PayloadHasher.ComputeHash` + comparaison ; `catch CryptographicException`) — testé |
| INV-STAGING-004 | Une entrée ABSENTE à la lecture est TRANSITOIRE (`StagedPayloadNotFoundException` : « pas encore stagé / à re-tenter »), JAMAIS un état terminal ni une perte | `FileSystemPayloadStagingStore.ReadAsync` (`File.Exists` → `StagedPayloadNotFoundException.ForKey`) — testé |
| INV-STAGING-005 | Isolation tenant : la clé `StagedPayloadKey` est tenant-scopée et le chemin inclut un segment de tenant assaini ; un tenant ne lit/n'existe/ne purge JAMAIS l'entrée d'un autre (même `document_id`) | `StagedPayloadKey` (tenant obligatoire) ; `FileSystemPayloadStagingStore.ResolveFullPath` (`StagingPathLayout.SanitizeSegment(tenant)` + vérification de périmètre) — testé 2 tenants |
| INV-STAGING-006 | Le magasin est PURGEABLE (magasin transitoire de traitement) : `PurgeAsync` supprime l'entrée, idempotent (no-op si absente). Purge LÉGITIME — NI table d'audit NI coffre WORM (CLAUDE.md n°4/12 inchangés) | `FileSystemPayloadStagingStore.PurgeAsync` (`File.Delete` conditionnel) — testé |
| INV-STAGING-007 | La purge après émission est SUBORDONNÉE à la présence EFFECTIVE du paquet WORM (`IArchiveStore.ExistsAsync` via `IArchivedDocumentProbe`), JAMAIS à la seule étiquette `Issued` : entre la transition `Issued` et l'écriture WORM (TRK05), le staging est CONSERVÉ | `StagingPurgeService.PurgeIfArchivedAsync` (purge uniquement si `IsArchivedAsync` vrai ; sinon conservé) — testé (WORM présent/absent) |
| INV-STAGING-008 | L'écriture est DURABLE avant le retour (flush disque) et idempotente sur la clé (ré-écriture du même contenu = no-op logique) ; jamais de fichier partiel au chemin canonique | `FileSystemPayloadStagingStore.WriteAsync` (`FileOptions.WriteThrough` + `Flush(flushToDisk)` + renommage atomique `File.Move(overwrite)`) — testé |
| INV-STAGING-009 | Le comportement est piloté par les capacités DÉCLARÉES (`PayloadStagingStoreCapabilities`), jamais par un test de type concret (`if (store is …)` interdit — P1, CLAUDE.md n°14) | `IPayloadStagingStore.Capabilities` ; `FileSystemPayloadStagingStore.Capabilities => None` (chiffrement/purge produit, indépendants du backend) |
