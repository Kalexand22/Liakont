# Invariants — module Supervision

| ID | Rule | Enforcement |
|---|---|---|
| INV-SUPERVISION-001 | C'est la PLATEFORME qui détecte l'absence (dead-man's switch, F12 §5.1) : l'évaluation est périodique (job système), JAMAIS pilotée par un événement de l'agent — détecter une absence ne peut pas reposer sur la réception d'un message. | Job système `SupervisionEvaluationTrigger` + `SupervisionEvaluationFanOutHandler` (planifié par le module Job) ; aucun `IIntegrationEventConsumer` dans le module ; tests `SupervisionJobTests`. |
| INV-SUPERVISION-002 | Le balayage multi-tenant passe EXCLUSIVEMENT par `TenantJobRunner` (SOL06) — aucune boucle « pour chaque tenant » maison (module-rules §6). C'est le SEUL code cross-tenant du produit, et il est en LECTURE (chaque tenant est évalué dans sa propre base via une connexion scopée). | `SupervisionEvaluationFanOutHandler` appelle `ITenantJobRunner.RunForAllTenantsAsync(new SupervisionEvaluationTenantJob())` ; `SupervisionEvaluationTenantJob` résout le moteur depuis `TenantJobContext.Services` (base du tenant) ; tests `SupervisionJobTests.FanOutHandler_*`, `DeadMansSwitchMultiTenantIntegrationTests`. |
| INV-SUPERVISION-003 | Anti-bruit : AU PLUS UNE alerte active (`resolved_utc IS NULL`) par règle dans la base du tenant. Une alerte active ne se RE-déclenche pas tant qu'elle n'est pas résolue. | Moteur `AlertEvaluationService` (n'insère que si `FindActiveByRuleAsync` est nul) + index unique partiel `ux_supervision_alerts_active_rule (rule_key) WHERE resolved_utc IS NULL` (`V002`) + `INSERT … ON CONFLICT DO NOTHING` ; tests `AlertEvaluationServiceTests.Firing_Twice_Raises_Once`, `AlertAntiNoiseIntegrationTests`. |
| INV-SUPERVISION-004 | Auto-résolution : quand la condition d'une règle disparaît, l'alerte active de cette règle est RÉSOLUE (`resolved_utc` renseigné). Après résolution, la même condition peut re-déclencher une NOUVELLE alerte. | Moteur `AlertEvaluationService.ApplyAsync` (résout l'alerte active si la règle ne se déclenche plus) + `Alert.Resolve` (lève si déjà résolue) ; tests `AlertEvaluationServiceTests.Clear_Resolves_Active`, `..._Refires_After_Resolution`, `AlertAntiNoiseIntegrationTests`. |
| INV-SUPERVISION-005 | Une alerte est de l'état OPÉRATIONNEL MUTABLE (résolution + acquittement) — PAS une table d'audit append-only. Les écritures sont CIBLÉES PAR OPÉRATION : `ResolveAsync` n'écrit que `resolved_utc` (sous garde `resolved_utc IS NULL`), `AcknowledgeAsync` n'écrit que `acknowledged_by`/`acknowledged_utc` ; aucune des deux ne réécrit les colonnes de l'autre, donc PAS de perte de mise à jour concurrente (lost-update) entre l'auto-résolution et l'acquittement. Le déclenchement, le tenant, la règle et la gravité ne changent jamais après création. | `PostgresAlertStore.ResolveAsync`/`AcknowledgeAsync` (UPDATE ciblés, colonnes possédées uniquement) ; `Alert` n'expose que `Resolve`/`Acknowledge` comme mutations ; tests `AlertTests`, `AlertLifecycleIntegrationTests`, `AlertConcurrencyIntegrationTests` (snapshot périmé dans les deux sens). |
| INV-SUPERVISION-006 | Acquitter n'affecte PAS la résolution : c'est un marqueur « prise en charge » (identité opérateur obligatoire), journalisé ; une alerte acquittée reste active tant que la condition persiste, puis se résout normalement. Acquitter une alerte absente retourne `false` (jamais d'exception, jamais de mutation silencieuse). | `Alert.Acknowledge` (identité requise, n'écrit pas `resolved_utc`) + `AlertAcknowledgementService.AcknowledgeAsync` (retourne `false` si absente) ; tests `AlertTests.Acknowledge_*`, `AlertLifecycleIntegrationTests.Acknowledge_*`. |
| INV-SUPERVISION-007 | L'échec d'évaluation d'une RÈGLE est isolé (les autres règles continuent) et REMONTÉ — jamais avalé silencieusement (un échec silencieux de supervision serait une panne silencieuse, l'inverse du but). L'échec d'un TENANT est isolé par le runner (les autres tenants continuent) et journalisé. | `AlertEvaluationService` (try/catch par règle → `AlertEvaluationResult.Failures`) ; `SupervisionEvaluationTenantJob` lève si des règles ont échoué (le runner l'isole et le remonte) ; `SupervisionEvaluationFanOutHandler` journalise les échecs par tenant ; tests `AlertEvaluationServiceTests.Failing_Rule_Is_Isolated`, `SupervisionJobTests`. |
| INV-SUPERVISION-008 | Isolation tenant PAR LA CONNEXION (database-per-tenant, blueprint §7) : les alertes vivent dans la base du tenant évalué, aucune requête cross-tenant en écriture. La colonne `tenant_id` est l'exception documentée à « aucune colonne de tenant » (étiquetage pour le dashboard cross-tenant en lecture seule, SUP02). | Store et lectures via `IConnectionFactory` (tenant courant) ; `tenant_id` renseigné depuis `TenantJobContext.TenantId` ; tests `DeadMansSwitchMultiTenantIntegrationTests` (isolation entre 2 bases). |
| INV-SUPERVISION-009 | Aucune règle fiscale ni seuil inventé dans SUP01a : le module ne livre AUCUNE règle concrète (au plus une règle d'exemple côté tests). Les seuils (F12 §5.2) et la cadence d'échéance (uniquement si `reportingFrequency` est renseigné) sont la responsabilité de SUP01b — jamais devinés (CLAUDE.md n°2). | Aucune `IAlertRule` enregistrée dans `SupervisionModuleRegistration` ; le moteur sans règle ne produit aucune alerte ; tests `AlertEvaluationServiceTests.No_Rules_Produces_No_Alert`. |
