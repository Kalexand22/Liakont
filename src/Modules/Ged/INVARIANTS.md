# Invariants — module Ged

Jeu d'invariants **consolidé** du module GED (F19 §8). L'item **GED02** (scaffold) **établit et publie** ce
jeu ; l'**enforcement** de chaque règle est porté par l'item cité en colonne *Enforcement* (la plupart
arrivent aux items GED03+). Les invariants d'architecture `INV-ARCH-GED-*` (coffre/WORM) complètent le jeu.

| ID | Rule | Enforcement |
|---|---|---|
| INV-GED-01 | **Anti-EAV** : un lien porte **exactement une colonne de valeur TYPÉE** (jamais un `value text` fourre-tout) ; toute donnée interrogeable est un **axe déclaré**. `value_number` est `numeric` (decimal C#), échelle portée par `value_scale` de l'axe (F19 §3.4.3/§3.5, règle 1). | `ValueNormalizer` (decimal half-up, colonne typée par `data_type` — **GED03a**, Domain, tests unitaires) ; `ged_index.document_axis_links` (colonnes typées + `ck_dal_value_or_retraction`, vue `current_axis_links` — **GED03b**) ; test `INV-GED-01` (base réelle, GED03b). GED02 : documenté (schémas vides). |
| INV-GED-02 | **Liens append-only PUR** : `document_axis_links`, `entity_relations`, `document_entity_links` et les change-logs n'acceptent **ni UPDATE ni DELETE ni TRUNCATE** (tout rôle). La **révision** se fait par **chaînage** `supersedes_id` (+ `is_retraction`), **jamais** par mutation d'une ligne (aucune colonne mutable sous append-only). | Triggers append-only (calqués `reject_archive_entry_mutation`) + vues `current_*` (**GED03b/GED03c**) ; tests append-only + rétractation (RL-24, base réelle). GED02 : documenté. |
| INV-GED-03 | **Mono-valeur Domain sous concurrence** : deux écritures simultanées d'un axe **mono-valeur** ⇒ **une seule** valeur courante (`current_axis_links` renvoie 1 ligne). Un test **séquentiel** est un faux-vert et ne suffit pas. | `SetAxisValueCommand` : `pg_advisory_xact_lock` / `SELECT … FOR UPDATE` dans la MÊME transaction avant de superséder (**GED04**, RL-02) ; test **concurrent** obligatoire. |
| INV-GED-04 | **`attributes` jsonb présentation-only** : jamais recherché, facetté ni traversé (garde-fou anti-EAV — toute donnée interrogeable est un axe déclaré). | `entity_instances.attributes` (**GED03c**) ; recherche/graphe ne lisent que les axes / `search_vector` (**GED08**) ; test `INV-GED-04`. |
| INV-GED-05 | **DEFER, jamais deviner** : un `documentType` **sans profil**, ou un axe **obligatoire non résolu**, range le document en `deferred` (visible console) — **jamais** mappé au hasard ni rejeté en silence (règle 3). | `GedMapper.Map` → `MappedDocument` **ou** DEFER (**GED12**) ; consommateur d'ingestion (**GED05b**) ; goldens DEFER. |
| INV-GED-06 | **Anti-doublon GED par hash, canal DISJOINT** : l'anti-doublon GED réutilise la primitive `PayloadHasher` mais sur un **canal séparé** (`ged_ingestion.ged_received_documents`, `UNIQUE (tenant_id, payload_hash)`) ; la logique `GedIngestionDecision` (3 cas) est **RE-COPIÉE** dans `Ged.Domain` (PAS une référence à `Ingestion.Domain`, RL-01). Aucune ligne `documents.documents`, aucun état fiscal. | Registre GED en base système (**GED05b**, F19 §4.3.1) ; golden `GedIngestionDecision` (RL-01). GED02 : schéma `ged_ingestion` vide. |
| INV-GED-07 | **WORM-neutralité (intégrité produit souveraine)** : un document GED-seul est rangé write-once **hors** chaîne fiscale ; le hash d'un paquet **facture** est **inchangé** après l'ajout de la surface GED (`IGenericArchiveService` / `IndexAxes`). La GED ne partage pas la chaîne de hashes fiscale. | Surface additive `Archive.Contracts` hash-neutre (**GED07**) ; test golden hash-neutralité facture (P1, base réelle). Voir INV-ARCH-GED-1/2. |
| INV-GED-08 | **Tenant-scope par la connexion** : liens, recherche, facettes, graphe, backfill — tout est scopé par la connexion du tenant ; **aucune** requête cross-tenant ; **aucune** jointure SQL cross-schéma `ged_* → documents.`/`mandats.`/`tvamapping.` (soft-link logique seulement). | `IConnectionFactory` (base tenant) / `ISystemConnectionFactory` (registre système, scopé) ; garde **lint SQL cross-schéma** self-testée (**GED11**) ; test isolation cross-tenant (base réelle). |
| INV-GED-09 | **Graphe borné ET bidirectionnel** : la traversée est bidirectionnelle, **bornée en profondeur** (anti-DoS), anti-cycle, paginée **keyset** (jamais OFFSET / chargement-tout) ; la confidentialité est héritée des `entity_types` aux extrémités. | Traversée `current_entity_relations` bornée (**GED08**) ; pages `/ged/objet` bornées côté SQL (**GED09c**) ; test `INV-GED-09`. |
| INV-GED-10 | **Confidentialité — masquage server-side** : un axe/une entité `is_confidential` n'est ni affiché, ni facetté, ni traversé, ni exporté, ni journalisé en clair sans le droit `liakont.ged.confidential` ; le **prédicat est MATÉRIALISÉ dans le SQL** (anti-oracle, RL-31). Chiffrement au repos = D9 **ouvert** (jamais affirmé livré). | Permission dédiée (**GED06**) ; prédicat SQL matérialisé recherche/facette/graphe (**GED08**) ; masquage du `consultation_log` (**GED13**) ; test masquage lecture/facette/graphe/export/log. |
| INV-GED-11 | **`consultation_log` append-only en base TENANT** : le journal de consultation (search / view / explore / export / open_archive) vit en **base tenant** (`IConnectionFactory`, **JAMAIS** `ISystemConnectionFactory` — sinon fuite cross-tenant dans l'audit socle partagé), append-only (UPDATE/DELETE/TRUNCATE rejetés). | `ged_index.consultation_log` + triggers + `IConsultationAuditWriter` (**GED13**, ADR-0036) ; test append-only (base réelle). |
| INV-GED-12 | **Généricité produit (testée)** : **aucun** littéral de vocabulaire métier (`lot`, `vente`, `pv`, `encheres`, `adjudication`, `acheteur`, `bordereau`) dans `src/Modules/Ged/**` hors tests/seed ; les démos (enchères ET BTP) sont **intégralement reconstructibles** depuis `deployments/` sans toucher le code. | Garde **lint anti-littéral** self-testée dans verify-fast ET ci.yml (**GED11**, RL-27) ; seeds fictifs `deployments/<demo>/` (**GED10**). |
| INV-ARCH-GED-1 | **Option C** : un document GED-seul est rangé **write-once (WORM)** via `IArchiveStore` (aucun chemin d'update/delete) mais **n'entre pas** dans la chaîne de hashes fiscale et **ne modifie ni n'étend** `documents.archive_entries` / `ArchiveVerifier`. La hash-neutralité des factures est **structurelle** (la GED ne partage pas la chaîne fiscale). | Rangement `_ged/` via `IArchiveStore`, aucune écriture `archive_entries` (**GED07**) ; test golden : aucune migration/surface GED ne touche la table ni le chemin fiscal. |
| INV-ARCH-GED-2 | **Intégrité de référence par nature de document (P1)** : pour une **facture**, la valeur probante de référence reste **chaîne de hashes + ancrage RFC 3161** (souveraine, inchangée) ; pour un **document GED**, l'intégrité locale de référence est le **rangement write-once WORM** + `content_hash` indexé (ce hash n'est qu'une **copie** indexée des octets write-once). Toute valeur probante renforcée est **déférée** à un fast-follow et **jamais affirmée** tant que non livrée. | `IArchiveVerifier` (chaîne + ancrage) réservé au GED **fiscal-lié** (`archive_entry_id` non nul) ; intégrité GED = re-lecture `IArchiveStore` vs `content_hash` (**GED07/GED09b**). |
| INV-ARCH-GED-3 | **NF Z42-013 jamais affirmée sur simple déclaration (P1)** : `ClaimsNfZ42013 == true` (déclaratif fournisseur) **n'autorise jamais à lui seul** une notice « conforme NF Z42-013 ». Seul `SealVerified` **+** une spécification de vérification autonome ratifiée (D1) le permet ; tant que D1 non tranché : au plus « scellé chez un tiers déclarant ». | `ISealedArchiveProvider` à capacités + vérifieur (**GED20**, fast-follow BLOCKED sur D1) ; jamais de notice de conformité sans `SealVerified`. |
