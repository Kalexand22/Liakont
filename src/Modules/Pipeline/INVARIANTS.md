# Pipeline Module — Invariants

| ID | Rule | Enforcement |
|---|---|---|
| INV-PIPELINE-001 | Le lecteur canonique est le MIROIR EXACT du writer du contrat : `Serialize(Read(json)) == json` octet par octet, échelle `decimal` préservée, énumérations par nom, dates `yyyy-MM-dd`, optionnels omis, collections toujours présentes | `PivotCanonicalJsonReader` (parse du texte brut des décimaux, `Enum.Parse` par nom) — testé `PivotCanonicalJsonReaderTests` (round-trip sur les 8 golden de document unique) |
| INV-PIPELINE-002 | Le lecteur NE RE-SÉRIALISE PAS pour ré-hacher : la re-vérification du `payload_hash` est faite par `IPayloadStagingStore.ReadAsync` sur la string brute (PIP00) | `PivotCanonicalJsonReader.Read` (pure désérialisation, sans hash) ; doc de classe |
| INV-PIPELINE-003 | Aucune règle fiscale inventée : le triplet {catégorie, taux, VATEX} vient TOUJOURS de la table validée du tenant (moteur `TvaMapper`). CHECK ne devine jamais adjudication/frais : il fournit `TvaMappingPart.Autre` (« hors du découpage » — fidèle à l'absence de cette distinction dans le pivot générique, ADR-0004/F03 §2.3) | Revue (P1, CLAUDE.md n°2) ; `CheckTvaMapping` (doc + `LinePart = Autre`) — testé `CheckTvaMappingTests` |
| INV-PIPELINE-008 | CHECK est SÛR PAR DÉFAUT : un régime/part absent de la table validée, une table absente, une ligne hors forme V1 (≠ 1 code régime ↔ 1 ventilation) ou une anomalie de validation BLOQUANTE ⇒ document `Blocked`, JAMAIS d'envoi à l'aveugle ni d'association régime→ventilation devinée | `DocumentReceivedConsumer` + `CheckTvaMapping.Evaluate` — testé `DocumentReceivedConsumerTests` / `CheckTvaMappingTests` |
| INV-PIPELINE-009 | GARDE-FOU PRODUCTION (PIP01b §3) : si un compte PA ACTIF est en environnement « Production » ET la table TVA n'est PAS validée ⇒ TOUS les documents restent `Blocked`. Jamais affaiblie en Warning (P1) | `DocumentReceivedConsumer.IsProductionContextAsync` + `mapping.IsValidated` — testé `DocumentReceivedConsumerTests` ; CLAUDE.md n°3 |
| INV-PIPELINE-010 | CHECK est IDEMPOTENT (livraison at-least-once) : il n'agit que sur un document encore `Detected` (rejeu d'un document avancé = no-op). Un contenu pas encore stagé (`StagedPayloadNotFoundException`) est TRANSITOIRE (ADR-0014) : propagé pour re-livraison, JAMAIS un blocage terminal | `DocumentReceivedConsumer` (pré-contrôle `Detected` ; `StagedPayloadNotFound` re-levée) — testé `DocumentReceivedConsumerTests` |
| INV-PIPELINE-011 | Les motifs de blocage sont persistés dans la piste d'audit APPEND-ONLY du module Documents via `IDocumentLifecycle.BlockAsync` ; CHECK ne fait aucun update/delete d'audit, et toute transition est tenant-scopée | `DocumentReceivedConsumer` (transitions via `IDocumentLifecycle` uniquement) ; CLAUDE.md n°4/12 |
| INV-PIPELINE-012 | Une exécution CHECK consigne UNE trace clôturée (`PipelineRunType.Check`, `PipelineRunTrigger.Event`) dans `pipeline.run_logs` du tenant ; les exécutions transitoires (non stagé / tenant non configuré) n'écrivent pas de trace (exécution non clôturée) | `DocumentReceivedConsumer` (`RunLog.Start/Complete` + `IPipelineRunLogStore.SaveAsync`) — testé `DocumentReceivedConsumerTests` |
| INV-PIPELINE-013 | Un contenu stagé altéré/illisible (`StagedPayloadIntegrityException` — empreinte ≠ attendue OU blob indéchiffrable) est PERSISTANT : CHECK BLOQUE le document avec un motif (visible opérateur), JAMAIS un dead-letter silencieux laissant le document figé en `Detected` (« bloquer plutôt qu'envoyer faux ») | `DocumentReceivedConsumer` (catch `StagedPayloadIntegrityException` → `BlockAndLogAsync`) — testé `DocumentReceivedConsumerTests` ; CLAUDE.md n°3 |
| INV-PIPELINE-004 | Le module n'accède aux autres modules que par leurs `Contracts` (jamais `Domain`/`Application`/`Infrastructure`), et ne référence aucune PA concrète | Références de projet (`Infrastructure.csproj`) ; module-rules §3 ; revue (P1, CLAUDE.md n°14) |
| INV-PIPELINE-005 | Le journal d'exécutions `pipeline.run_logs` est TENANT-SCOPÉ par la connexion (database-per-tenant) ; aucune lecture cross-tenant | `PostgresPipelineRunQueries` (via `IConnectionFactory`) ; CLAUDE.md n°9/17 |
| INV-PIPELINE-006 | Une exécution (`RunLog`) ne peut pas se clôturer avant son début, ni porter un compteur négatif | `RunLog.Complete` (gardes `completedAt >= StartedAt`, compteurs ≥ 0) — testé `RunLogTests` |
| INV-PIPELINE-007 | `run_type` / `run_trigger` sont persistés par NOM d'énumération et relus tels quels (fidélité base ↔ contrat) | `PostgresPipelineRunQueries.MapRun` (`Enum.Parse`) — testé `PipelineRunLogQueriesIntegrationTests` |
| INV-PIPELINE-014 | SEND est un job PAR TENANT (`ITenantJob` exécuté par `ITenantJobRunner`, SOL06) ; AUCUNE boucle multi-tenant locale, et « un seul job send/sync par tenant à la fois » est garanti par l'ordonnanceur (pas de mutex applicatif) | `SendAllFanOutHandler` (`RunForAllTenantsAsync`) + `SendTenantJob` — testé `SendAllFanOutHandlerTests` ; CLAUDE.md n°9 |
| INV-PIPELINE-015 | DIAGNOSTIC PA (PIP01c §3, F04 §3.1, « Transport not available ») : SIREN non publié / `tax_report_setting` inactif (`StartDate` absente ou future) ⇒ AUCUN envoi (documents maintenus `ReadyToSend`) + Warning. Jamais d'envoi à l'aveugle | `SendTenantJob.IsTaxReportSettingActive` — testé `SendTenantJobTests` / `SendTenantJobIntegrationTests` ; CLAUDE.md n°3 |
| INV-PIPELINE-016 | ANTI-DOUBLON inter-traitement (TRK03, F05) : avant tout renvoi d'un `Sending` (crash) ou retry d'un `TechnicalError`, si le document porte une référence PA et que la PA le déclare `Issued` ⇒ `MarkIssued` SANS renvoyer ; sinon le renvoi est dédoublonné par la PA (numéro de document) — JAMAIS une double émission | `SendTenantJob.TryFinalizeFromPaStatusAsync` + idempotence PA (F05) — testé `SendTenantJobTests` / `SendTenantJobIntegrationTests` |
| INV-PIPELINE-017 | ÉMISSION : sur `Issued` ⇒ `MarkIssued` (preuve) + archive WORM (TRK05) ; la PURGE du staging est SUBORDONNÉE à la présence EFFECTIVE du paquet WORM (`IStagingPurgeService` → `IArchivedDocumentProbe` → `IArchiveStore.ExistsAsync`), JAMAIS à la seule étiquette `Issued` (entre la transition et l'écriture WORM = staging CONSERVÉ) | `SendTenantJob.FinalizeIssuedAsync` (`PurgeIfArchivedAsync`) — testé `SendTenantJobIntegrationTests` (Issued + WORM absent = conservé) ; ADR-0014 §4 |
| INV-PIPELINE-018 | REJET PA (`RejectedByPa`) et ERREUR TECHNIQUE (`TechnicalError`) CONSERVENT le staging (contenu requis pour correction/resoumission, jamais archivé en WORM) ; aucune purge sur rejet/erreur | `SendTenantJob.HandleSendResultAsync` — testé `SendTenantJobTests` / `SendTenantJobIntegrationTests` ; ADR-0014 |
| INV-PIPELINE-019 | La preuve d'échange PA (`pa_response_snapshot`, colonne `jsonb`) est TOUJOURS du JSON valide : la réponse brute de la PA — potentiellement non-JSON — est EMBARQUÉE dans un champ chaîne `rawResponse`, jamais réinterprétée | `SendPaSnapshot` (`JsonSerializer`) — testé `SendTenantJobIntegrationTests` (rejet PA en texte brut archivé) |
| INV-PIPELINE-020 | DRY-RUN (PIP01c §5) : « tout sauf écritures PA » — dénombre les `ReadyToSend`, n'appelle AUCUNE écriture PA (`SendDocumentAsync`/`Ensure…`) et ne fait avancer aucun document ; un `RunLog` SEND est tout de même consigné | `SendTenantJob.ExecuteAsync` (branche dry-run) — testé `SendTenantJobTests` |
| INV-PIPELINE-021 | Un avoir (`CreditNoteRefs` non vide) vers une PA sans capacité avoirs reste `ReadyToSend` (traité par le pipeline des avoirs, PIP02), JAMAIS bloqué ni envoyé à l'aveugle (l'état machine interdit `Sending → ReadyToSend`) | `SendTenantJob.SendReadyAsync` (garde capacité avant `BeginSending`) ; piloté par `PaCapabilities.SupportsCreditNotes` |
| INV-PIPELINE-022 | SYNC (PIP01d) est un job PAR TENANT (`ITenantJob`/`ITenantJobRunner`, SOL06) de RÉCONCILIATION en lecture seule : pour chaque `Issued`, récupère la facture PA générée (SI `SupportsDocumentRetrieval`) et le(s) tax report(s) (SI `SupportsTaxReportRetrieval`) → ADDENDA WORM (TRK05). Piloté par les CAPACITÉS déclarées, jamais par `if (pa is …)` ni un flag produit | `SyncTenantJob` + `SyncAllFanOutHandler` — testé `SyncTenantJobIntegrationTests` (avec/sans capacités) ; CLAUDE.md n°8/16 |
| INV-PIPELINE-023 | ATTRIBUTION TAX REPORT — PAR DOCUMENT, jamais inventée : un tax report n'est rattaché à un document que s'il figure dans les `tax_report_ids` que la PA renvoie POUR CE document (`GetDocumentStatusAsync`) ; le contenu vient de `ListTaxReportsAsync`. Les addenda sont IDEMPOTENTS (adressage par empreinte de contenu) — un SYNC ré-exécuté ne duplique jamais | `SyncTenantJob.SyncDocumentAsync` + `IArchiveService.AddAddendumAsync` (chemin dérivé du hash) — testé `SyncTenantJobIntegrationTests` (idempotence) ; CLAUDE.md n°2/4 |
| INV-PIPELINE-024 | SYNC ne fait avancer AUCUNE machine à états, n'écrit RIEN dans la base source et n'effectue AUCUNE transmission ; un tax report dont le XML n'est pas encore disponible est repris au cycle suivant (jamais un échec), un XML base64 illisible est ignoré (jamais un addendum corrompu archivé) | `SyncTenantJob` (addenda seuls ; `Convert.FromBase64String` gardé) ; coffre WORM append-only inchangé |
