# Pipeline Module — Invariants

| ID | Rule | Enforcement |
|---|---|---|
| INV-PIPELINE-001 | Le lecteur canonique est le MIROIR EXACT du writer du contrat : `Serialize(Read(json)) == json` octet par octet, échelle `decimal` préservée, énumérations par nom, dates `yyyy-MM-dd`, optionnels omis, collections toujours présentes | `PivotCanonicalJsonReader` (parse du texte brut des décimaux, `Enum.Parse` par nom) — testé `PivotCanonicalJsonReaderTests` (round-trip sur les 8 golden de document unique) |
| INV-PIPELINE-002 | Le lecteur NE RE-SÉRIALISE PAS pour ré-hacher : la re-vérification du `payload_hash` est faite par `IPayloadStagingStore.ReadAsync` sur la string brute (PIP00) | `PivotCanonicalJsonReader.Read` (pure désérialisation, sans hash) ; doc de classe |
| INV-PIPELINE-003 | Aucune règle fiscale inventée : le triplet {catégorie, taux, VATEX} vient TOUJOURS de la table validée du tenant (moteur `TvaMapper`). CHECK ne devine jamais adjudication/frais : il fournit `TvaMappingPart.Autre` (« hors du découpage » — fidèle à l'absence de cette distinction dans le pivot générique, ADR-0004/F03 §2.3) | Revue (P1, CLAUDE.md n°2) ; `CheckTvaMapping` (doc + `LinePart = Autre`) — testé `CheckTvaMappingTests` |
| INV-PIPELINE-008 | CHECK est SÛR PAR DÉFAUT : un régime/part absent de la table validée, une table absente, une ligne hors forme V1 (≠ 1 code régime ↔ 1 ventilation) ou une anomalie de validation BLOQUANTE ⇒ document `Blocked`, JAMAIS d'envoi à l'aveugle ni d'association régime→ventilation devinée | `DocumentReceivedConsumer` + `CheckTvaMapping.Evaluate` — testé `DocumentReceivedConsumerTests` / `CheckTvaMappingTests` |
| INV-PIPELINE-009 | GARDE-FOU PRODUCTION (PIP01b §3) : si un compte PA ACTIF est en environnement « Production » ET la table TVA n'est PAS validée ⇒ TOUS les documents restent `Blocked`. Jamais affaiblie en Warning (P1) | `DocumentReceivedConsumer.IsProductionContextAsync` + `mapping.IsValidated` — testé `DocumentReceivedConsumerTests` ; CLAUDE.md n°3 |
| INV-PIPELINE-010 | CHECK est IDEMPOTENT (livraison at-least-once) : il n'agit que sur un document encore `Detected` (rejeu d'un document avancé = no-op). Un contenu pas encore stagé (`StagedPayloadNotFoundException`) est TRANSITOIRE (ADR-0014) : propagé pour re-livraison, JAMAIS un blocage terminal | `DocumentReceivedConsumer` (pré-contrôle `Detected` ; `StagedPayloadNotFound` re-levée) — testé `DocumentReceivedConsumerTests` |
| INV-PIPELINE-011 | Les motifs de blocage sont persistés dans la piste d'audit APPEND-ONLY du module Documents via `IDocumentLifecycle.BlockAsync` ; CHECK ne fait aucun update/delete d'audit, et toute transition est tenant-scopée | `DocumentReceivedConsumer` (transitions via `IDocumentLifecycle` uniquement) ; CLAUDE.md n°4/12 |
| INV-PIPELINE-012 | Une exécution CHECK consigne UNE trace clôturée (`PipelineRunType.Check`, `PipelineRunTrigger.Event`) dans `pipeline.run_logs` du tenant ; les exécutions transitoires (non stagé / tenant non configuré) n'écrivent pas de trace (exécution non clôturée) | `DocumentReceivedConsumer` (`RunLog.Start/Complete` + `IPipelineRunLogStore.SaveAsync`) — testé `DocumentReceivedConsumerTests` |
| INV-PIPELINE-013 | Un contenu stagé altéré/illisible (`StagedPayloadIntegrityException` — empreinte ≠ attendue OU blob indéchiffrable) est PERSISTANT : CHECK BLOQUE le document avec un motif (visible opérateur), JAMAIS un dead-letter silencieux laissant le document figé en `Detected` (« bloquer plutôt qu'envoyer faux ») | `DocumentReceivedConsumer` (catch `StagedPayloadIntegrityException` → `BlockAndLogAsync`) — testé `DocumentReceivedConsumerTests` ; CLAUDE.md n°3 |
| INV-PIPELINE-004 | Le module n'accède aux autres modules que par leurs `Contracts` (jamais `Domain`/`Application`/`Infrastructure`), et ne référence aucune PA concrète | Références de projet (`Infrastructure.csproj`) ; module-rules §3 ; revue (P1, CLAUDE.md n°14) |
| INV-PIPELINE-005 | Le journal d'exécutions `pipeline.run_logs` est TENANT-SCOPÉ par la connexion (database-per-tenant) ; aucune lecture cross-tenant | `PostgresPipelineRunQueries` (via `IConnectionFactory`) ; CLAUDE.md n°9/17 |
| INV-PIPELINE-006 | Une exécution (`RunLog`) ne peut pas se clôturer avant son début, ni porter un compteur négatif | `RunLog.Complete` (gardes `completedAt >= StartedAt`, compteurs ≥ 0) — testé `RunLogTests` |
| INV-PIPELINE-007 | `run_type` / `run_trigger` sont persistés par NOM d'énumération et relus tels quels (fidélité base ↔ contrat) | `PostgresPipelineRunQueries.MapRun` (`Enum.Parse`) — testé `PipelineRunLogQueriesIntegrationTests` |
