# Invariants — module Mandats

> Source normative : ADR-0022 (Décision + Invariants), F15 §2. MND01 livre et prouve les invariants de
> structure ci-dessous ; les invariants d'acceptation/numérotation/gate (INV-ACCEPT-*, INV-BT1-*) sont
> portés par MND02-MND07.

| ID | Rule | Enforcement |
|---|---|---|
| INV-MANDATS-1 | Tout `Mandant`/`Mandat` **et le journal `mandat_change_log`** est **tenant-scopé** (`WHERE company_id = @CompanyId` ; `company_id` **NOT NULL** sur le journal) ; aucune requête cross-tenant ; un mandant n'est **jamais** un sous-tenant (blueprint §7, CLAUDE.md n°9/17). | `company_id` sur chaque table + clause `WHERE company_id` dans `PostgresMandatsUnitOfWork`, `MandatsMaterializer`, `PostgresMandatsQueries` ; clés `uq_mandants_company_reference` / `uq_mandats_company_mandant_reference` ; tests d'intégration d'isolation sur ≥ 2 sociétés. |
| INV-MANDATS-2 | Le module `Mandats` n'est accédé par les autres modules **que par ses `Contracts`** (NetArchTest) ; aucune logique Mandat dans l'agent. | `Contracts` ne référence que `Stratum.Common.Abstractions` ; `MandatsBoundaryTests` (scan déclaratif des `.csproj` + réflexion sur les assemblies référencées). |
| INV-MANDATS-3 | Registre et cycle de vie journalisés **append-only en transaction** (mutation + entrée de journal atomiques) ; immuabilité par **trigger base** (UPDATE/DELETE/TRUNCATE rejetés, message FR citant CLAUDE.md n°4) ; **aucun** chemin update/delete sur le journal côté repository. | `PostgresMandatsUnitOfWork` (Insert*/Save*Mutation insèrent toujours l'entrée dans la même `TransactionScope`) ; `V004__create_mandat_change_log_table.sql` (fonction + 2 triggers) ; tests `ChangeLog_Is_Append_Only_*` / `ChangeLog_Truncate_Is_Rejected` / `Mutation_And_ChangeLog_Are_Atomic_*`. |
| INV-MANDATS-4 | `AssujettissementStatus` `null` **OU** `ContestationDelay` `null` **OU** mandat non validé **OU** révoqué ⇒ autofacturation 389 **SUSPENDUE** ; **aucun** défaut inventé (CLAUDE.md n°2/3). | `Mandat.IsSelfBillingSuspended` (les 4 conditions) ; colonnes nullables `assujettissement_status` / `contestation_delay` ; tests cartésiens `Mandat_IsSelfBillingSuspended_*`. La **garde** d'émission (`ISelfBilledGate`) est livrée par MND03. |
| INV-MANDATS-5 | **Aucune** donnée client réelle dans le code ni les fixtures committées ; mandants/préfixes/n° TVA réels = paramétrage tenant ou `deployments/<client>/` ; seuls exemples fictifs (CLAUDE.md n°7). | Aucune valeur réelle dans `src/` ; fixtures de test exclusivement fictives (références « MANDANT-EXEMPLE-* », SIREN fictifs). |
| INV-MANDATS-6 | Une mutation de mandat repasse le mandat **« NON VALIDÉE »** (`Invalidate()`) ⇒ 389 suspendu jusqu'à revalidation humaine (gabarit `MappingTable`, ADR-0022 §3). | `Mandat.UpdateTerms` appelle `Invalidate()` ; `Mandat.Validate` exige une identité de valideur ; tests `Mandat_UpdateTerms_Invalidates` / `Mandat_Mutation_Invalidates_And_Logs`. |
| INV-MANDATS-7 | Le mandat ne porte **aucun montant** (`decimal`/`float` absent du modèle, ADR-0022) ; la numérotation par mandant (`NextValue` bigint) est **hors MND01** (MND05). | Aucun champ monétaire/`NextValue` dans `Mandat`/`Mandant` ni dans le schéma `mandats` (V002/V003). |
| INV-ACCEPT-1 | Le workflow d'acceptation est géré par le module générique **DocumentApproval** (purpose `SelfBilledAcceptance`, depuis SIG05 / ADR-0024 amendé par ADR-0028) ; `mandats.self_billed_acceptances` est une **companion fiscale** (allocated_number/pending_since) ; **aucun** état n'est ajouté à `DocumentState` (le blocage réutilise `Blocked` via un motif — port `ISelfBilledGate`, MND03). | `documentapproval.document_validations` (purpose 0, attempt 1) + companion `mandats.self_billed_acceptances` (V005 tronquée par V010) ; MND02 ne touche pas le module `Documents` (MandatsBoundaryTests). La **garde** d'émission est livrée par MND03. |
| INV-ACCEPT-2 | **Garde avant émission** (MND03, ADR-0024 §3) : un document self-billed n'est émissible que si son acceptation est dans `{Accepted, TacitlyAccepted}` ; sinon (en attente, contestée, ou **aucun** enregistrement) il est **maintenu `Blocked`** via un motif, **jamais** `Sent` — par **inversion de dépendance** (le pipeline n'interroge que l'abstraction `ISelfBilledGate`, jamais le module concret — frontière P1, CLAUDE.md n°14). **Aucun** nouvel état ajouté à `DocumentState`. | Port `ISelfBilledGate` (`Mandats.Contracts`) + impl `SelfBilledGate` (lit `IsAccepted`, fail-closed sur absence) ; interrogé par `DocumentCheckEvaluator` (source UNIQUE de la décision : CHECK + recheck + réconciliation des avoirs → un recheck post-acceptation rouvre le gate). Couverture tests : `SelfBilledGateTests` (Mandats) + `DocumentReceivedConsumerTests.SelfBilled_*` (CHECK initial, gate factice ouvert/fermé/contesté + non-régression) + `DocumentRecheckServiceTests.SelfBilled_*` (recheck, gate ouvert → ReadyToSend, gate fermé → stays Blocked). La réconciliation des avoirs (`SendTenantJob`) partage la même source de décision (`DocumentCheckEvaluator`) mais n'est pas couverte séparément par un test unitaire du cas self-billed : un avoir 261 auto-facturé y serait bloqué fail-closed (acceptation portée par son propre `document_id`, inexistante) — comportement SÛR, mais le passage d'un 261 par le gate est **NON TRANCHÉ** (F15 §6.5 item 5), chemin volontairement non testé jusqu'à l'arbitrage. |
| INV-ACCEPT-4 | Machine d'acceptation **FERMÉE** `PendingAcceptance → {Accepted, TacitlyAccepted, Contested}` ; toute transition hors graphe rejetée (produit cartésien) ; **aucun** retour arrière depuis un état terminal. Depuis SIG05 la machine est celle de **DocumentApproval** (sous-graphe purpose `SelfBilledAcceptance`, fermé par le workflow générique). | `DocumentApprovalWorkflow` (DocumentApproval.Infrastructure) ; `SelfBilledAcceptanceTests.Closed_Machine_Allows_Only_From_Pending` (cartésien 4 états × 3 transitions via `ISelfBilledAcceptanceCommands`). |
| INV-ACCEPT-5 | **Toute** transition (création incluse) écrit une entrée dans `documentapproval.document_approval_log` dans la **MÊME transaction** DocumentApproval (amendé SIG05 per ADR-0024 / F17 §4) ; journal immuable par trigger base (UPDATE/DELETE/TRUNCATE rejetés) et tenant-scopé (`company_id` NOT NULL). L'ancien journal `mandats.self_billed_acceptance_log` a été relocalisé par la migration V010 et supprimé. | `DocumentApprovalWorkflow.RequestValidationAsync` / `RecordRecordedValidationAsync` / etc. (genèse + journal atomiques en une transaction) ; triggers `trg_document_approval_log_append_only` / `_no_truncate` (DocumentApproval V003) ; tests `SelfBilledAcceptanceIntegrationTests.*Log_Is_Append_Only*` + `*_Atomic*`. |
| INV-BT1-1 (MND05) | En 389, `Number` (BT-1 du payload hashé) **reste rempli** par l'identifiant source ; le **BT-1 fiscal alloué** est une valeur SÉPARÉE, **hors payload hashé** (assignée à l'émission sur `self_billed_acceptances.allocated_number`). `CanonicalJson` n'est PAS modifié ; aucune branche de format au flux 389 (ADR-0007 préservé). | `CanonicalJson` inchangé ; `PostgresSelfBilledNumberAllocator.AssignAcceptanceSql` (écrit hors pivot) ; tests `CanonicalJsonRulesTests.SelfBilled_*` (Number = source, pas de numéro fiscal au canonique ; le booléen est le seul différenciateur). |
| INV-BT1-2 (MND05) | L'allocation du BT-1 fiscal est **idempotente** sur la clé source : une ré-extraction d'un même document source relit le **même** numéro, **jamais** ré-alloué (la séquence n'avance pas). | `PostgresSelfBilledNumberAllocator` (get-or-create sous verrou sur `mandat_number_allocations`, clé `(company_id, mandant_id, source_reference)` immuable — V009 + 2 triggers) ; tests `MandatNumberAllocator*` (`Idempotent_On_Source_Key`, `ReExtraction_*`, `Allocation_Registry_Is_Immutable_*`). |
| INV-BT1-4 (MND05) | La séquence est **par mandant** et **tenant-scopée** (`MandatSequence` clé `(company_id, mandant_id)`, verrou `FOR UPDATE` par mandant, jamais cross-tenant) ; `NextValue` en **`bigint`**, jamais float. « Chronologique » et « continue » (§1.4) sont distincts ; sous concurrence d'un même mandant, l'allocation est sérialisée (sans doublon ni trou). | `MandatSequence` (bigint, `Allocate()` avance de 1) + `V008__create_mandat_sequences_table.sql` ; `PostgresSelfBilledNumberAllocator.LockSequenceForUpdateSql` ; tests `MandatSequenceTests` (bigint, format, avancement) + `MandatNumberAllocator*` (`Concurrent_*_Serialized_Without_Duplicate`, `Sequences_Are_Independent_Per_Mandant`, `Allocation_Is_Tenant_Scoped`). |
| INV-BT1-3 (MND05+MND06) | **Garde d'ingestion 389 SUBSTITUÉE, pas affaiblie** (CLAUDE.md n°3) : un 389 sans clé d'idempotence source est **rejeté** ; en 389, l'anti-doublon ne se fie **jamais** au `Number` brut de la source (numéroté globalement par un legacy) mais à la clé `(mandant_id, BT-1 fiscal alloué)`. Le `Number` reste rempli (idempotence interne via `SourceReference`), ADR-0007 préservé. | **Ingestion** : `Document.CreateDetected` exige `source_reference` (test `CreateDetected_Requires_SourceReference`). **Allocation** : `PostgresSelfBilledNumberAllocator.AllocateAsync` rejette une `source_reference` vide (`ArgumentException`). **Dédup** (module Documents, MND06) : `PostgresDuplicateDocumentCheck` branche sur `MandantId` → clé 389 sur le BT-1 alloué, pas le numéro source ; tests `SelfBilled389AntiDuplicateIntegrationTests`. |
| INV-BT1-5 (MND06) | **Anti-doublon 389** : deux 389 de **même numéro** mais **mandants différents** → **Send** ; **même mandant** + ré-extraction (même BT-1 idempotent) → **Blocked** ; la clé `(supplier_siren, document_number)` **reste** appliquée aux **non-389** (aucune régression). | Re-clé dans **`DuplicateCheckRequest.MandantId`** (additif nullable, module Documents) + branche `PostgresDuplicateDocumentCheck` (clé `(mandant_id, document_number)` en 389, sinon historique) ; décision pure inchangée `DocumentDuplicatePolicy` ; tests `SelfBilled389AntiDuplicateIntegrationTests.*` (cartésien mandants/ré-extraction + parité §4.3 + non-régression non-389). |
| INV-BT1-6 (MND06) | La re-clé 389 est portée par un **index d'unicité SQL atomique**, **jamais** par une neutralisation applicative du SIREN (qui ouvrirait une fenêtre de doublon sur un 389 en attente ré-extrait). | Index PARTIEL `uq_documents_mandant_number_issued_389 (mandant_id, document_number) WHERE mandant_id IS NOT NULL AND state = 'Issued'` (`V010__add_mandant_id_and_unique_index_389.sql`, module Documents) — atomique au moment de l'émission, **partiel sur Issued** pour préserver le remplacement F06 §4.3 (INV-DOCUMENTS-006) ; tests `SelfBilled389AntiDuplicateIntegrationTests.Unique_Index_*`. |

> INV-ACCEPT-2 (gate avant émission) est livré par **MND03**, INV-ACCEPT-3/INV-ACCEPT-6 (bascule tacite) par
> **MND04**, INV-BT1-1/2/4 (allocation hybride) par **MND05**. INV-BT1-3 (garde d'ingestion 389 substituée) est
> désormais **COMPLET** (MND05 = rejet de la clé source vide à l'allocation ; MND06 = re-clé anti-doublon sur le
> BT-1 alloué, jamais le numéro source). INV-BT1-5/INV-BT1-6 (re-clé anti-doublon F06 + index d'unicité 389) sont
> livrés par **MND06**. NB : l'enforcement de la re-clé vit dans le module **Documents** (`DuplicateCheckRequest`,
> `PostgresDuplicateDocumentCheck`, migration `V010`) — c'est l'amendement de F06 §4 (cf. `Documents/INVARIANTS.md`
> INV-DOCUMENTS-012) ; la famille BT-1 est tracée ici par cohérence d'ADR (ADR-0025). La projection du type 389 +
> parties dans le payload PA sortant (émission RÉELLE de bout en bout) reste portée par **MND07**.
