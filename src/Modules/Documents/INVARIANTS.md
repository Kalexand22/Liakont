# Invariants — module Documents

| ID | Rule | Enforcement |
|---|---|---|
| INV-DOCUMENTS-001 | Les montants (`total_net`, `total_tax`, `total_gross`) sont en `decimal` exact, jamais flottant (CLAUDE.md n°1). Round-trip base ↔ decimal sans perte. Aucun montant à plus de 2 décimales n'est accepté (la colonne `numeric(18,2)` le tronquerait silencieusement — intégrité de stockage, CLAUDE.md n°4). | Colonnes `numeric(18,2)` (`V002`) ; propriétés `decimal` de `Document` ; garde-fou `RequireMonetaryScale` dans `Document.CreateDetected` ; tests `CreateDetected_RoundTrips_Decimal_Amounts_Without_Loss`, `CreateDetected_Rejects_Total*_With_More_Than_2_Decimals`, `CreateDetected_Accepts_Total*_With_Two_Or_Fewer_Decimals`. |
| INV-DOCUMENTS-002 | Le module ne calcule ni ne valide rien : il REPORTE les montants calculés par la source et le type de document BRUT (frontière module-rules §2 ; classification et contrôle des totaux délégués à Validation — ADR-0004 D3-3, F04). | `DetectedDocumentMapper.ToDetectedDocument` reporte `Totals.*` et `SourceDocumentKind` tels quels ; tests `DetectedDocumentMapperTests`. |
| INV-DOCUMENTS-003 | La création du document est IDEMPOTENTE sur l'identifiant (contrat `IDocumentIntake`) : un re-push n'ajoute ni document ni événement, et n'écrase pas un état déjà avancé. | `INSERT … ON CONFLICT (id) DO NOTHING` (`PostgresDocumentUnitOfWork.CreateDetectedAsync`, événement de genèse écrit seulement si insertion) ; tests `CreateDetected_Is_Idempotent_On_Id`, `Intake_Is_Idempotent_On_DocumentId`. |
| INV-DOCUMENTS-004 | Un document est créé en état `Detected` par l'ingestion (F06 §3, PIV04). | `Document.CreateDetected` fixe `State = Detected` ; tests `CreateDetected_Sets_State_Detected_And_Timestamps`, `Agent_Push_Creates_Detected_Document_With_Genesis_Event`. |
| INV-DOCUMENTS-005 | Tout champ source absent reste `null`, jamais un défaut implicite masquant une donnée manquante (blueprint §8). | `Document` : `supplier_siren` / `customer_name` / `pa_document_id` / `mapping_version` nullables ; `NullIfBlank` ; tests `CreateDetected_Blank_Siren_And_Customer_Become_Null`, `Maps_Null_Customer_*`, `Maps_Null_Supplier_Siren`. |
| INV-DOCUMENTS-006 | Le numéro de document n'est PAS une clé d'unicité en base : un document rejeté peut être remplacé sous un nouveau numéro, l'ancien passant `Superseded` (F06 §4). L'anti-doublon (état Issued / empreinte) est une règle APPLICATIVE (TRK03), jamais une contrainte qui casserait le remplacement. | Index NON unique `ix_documents_number` (`V002`) ; `IDocumentQueries.GetByNumberAsync` retourne le document le plus récent ; test `GetByNumber_Returns_Most_Recent_For_That_Number`. |
| INV-DOCUMENTS-007 | La piste d'audit `document_events` est APPEND-ONLY (CLAUDE.md n°4) : aucun chemin d'update/delete applicatif, et des triggers base REJETTENT tout UPDATE/DELETE d'une entrée ET tout TRUNCATE de la table — garantie indépendante du code. La table `archive_entries` est WORM (write-once) : chaque paquet ou addendum est une NOUVELLE ligne, TRK05 ne réécrit jamais une référence ; des triggers identiques (même discipline) ferment tout vecteur de mutation avant la livraison de TRK05. | `IDocumentUnitOfWork` n'expose que `AppendEventAsync` (aucun update/delete) ; triggers `trg_document_events_append_only` + `trg_document_events_no_truncate` (`V003`) + `trg_archive_entries_append_only` + `trg_archive_entries_no_truncate` (`V005`) ; tests `Update_…`, `Delete_…`, `Truncate_Of_The_Audit_Table_Is_Rejected`, `ArchiveEntry_Update_Is_Rejected`, `ArchiveEntry_Delete_Is_Rejected`. |
| INV-DOCUMENTS-008 | Isolation tenant PAR LA CONNEXION (database-per-tenant, blueprint §7) : le schéma vit dans la base DU TENANT, aucune colonne de tenant, aucune requête cross-tenant possible (CLAUDE.md n°9/17). | Lectures via `IConnectionFactory` (tenant courant, `ITenantContext`) ; ingestion via `ITenantConnectionFactory` par slug (`TenantSlugConnectionFactoryAdapter`) — la connexion est la frontière de tenant. |
