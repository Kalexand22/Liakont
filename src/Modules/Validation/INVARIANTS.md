# Validation Module — Invariants

| ID | Rule | Enforcement |
|---|---|---|
| INV-VALIDATION-001 | Une règle qui lève une exception ne fait JAMAIS passer le document : le pipeline produit une anomalie BLOQUANTE `RULE_CRASHED` (jamais de règle silencieuse — CLAUDE.md n°3, F04) | `ValidationPipeline.ValidateAsync` (try/catch → `CreateRuleCrashedIssue`, sévérité `Blocking`) ; testé `ValidationPipelineTests` |
| INV-VALIDATION-002 | Une règle en échec n'interrompt pas les autres : toutes les règles enregistrées s'exécutent, les anomalies sont agrégées | `ValidationPipeline.ValidateAsync` (boucle `foreach` avec catch par règle) ; testé `ValidationPipelineTests` |
| INV-VALIDATION-003 | Un document est `IsValid` si et seulement s'il ne porte AUCUNE anomalie bloquante ; une alerte (Warning) n'invalide pas (F04 §3.3) | `ValidationResult.IsValid` / `HasBlockingIssue` ; testé `ValidationResultTests`, `ValidationPipelineTests` |
| INV-VALIDATION-004 | Toute anomalie a un code stable non vide et un message opérateur en français non vide (CLAUDE.md n°12) | `ValidationIssue` (constructeur, `ArgumentException` sur code/message vide) ; testé `ValidationResultTests` |
| INV-VALIDATION-005 | Une annulation RÉELLE (token demandé) est propagée, jamais avalée ni convertie en `RULE_CRASHED` ; une `OperationCanceledException` PARASITE (timeout aval, token non annulé) est traitée comme un crash de règle (`RULE_CRASHED`), jamais laissée échapper | `ValidationPipeline.ValidateAsync` (`ThrowIfCancellationRequested` + `catch (OperationCanceledException) when (cancellationToken.IsCancellationRequested) → throw`, sinon `RULE_CRASHED`) ; testé `ValidationPipelineTests` |
| INV-VALIDATION-006 | Tout document validé est scopé à un tenant résolu (clé d'isolation, CLAUDE.md n°9) | `DocumentValidationContext` (constructeur rejette `Guid.Empty`) ; testé `DocumentValidationContextTests` |
| INV-VALIDATION-007 | Le module ne corrige ni ne normalise jamais les données : il détecte seulement (frontière `Validation`, `module-rules.md` §2) | Aucune écriture sur le document pivot (entrée en lecture seule) ; règle `IDocumentRule` retourne des anomalies, jamais un document modifié |
| INV-VALIDATION-008 | Un SIREN (9 chiffres) / SIRET (14 chiffres) est validé par la clé de Luhn ; le SIREN de La Poste (356000000) est autorisé par dérogation documentée — aucune règle d'identité inventée (F04 §4.1, CLAUDE.md n°2) | `SirenValidator` / `SiretValidator` (`Domain/Identity`) ; testé `SirenValidatorTests`, `SiretValidatorTests` |
| INV-VALIDATION-009 | Le validateur `FrenchVatNumberValidator` contrôle le format (« FR » + clé + SIREN) et la clé officielle `(12 + 3 × (SIREN mod 97)) mod 97` (F04 §4.2). C'est une **brique réutilisable** (consommée par VAL05 / contrôle de cohérence TVA à venir) : aucune règle bloquante VAL02 ne l'enforce — F04 §3 ne liste pas de contrôle de format de n° TVA. La cohérence « SIREN intégré = SIREN émetteur » (F04 §4.2) relèvera d'une règle dédiée, pas du validateur seul. | `FrenchVatNumberValidator` (`Domain/Identity`) ; testé `FrenchVatNumberValidatorTests` |
| INV-VALIDATION-010 | Un code pays est valide ssi il appartient aux 249 codes ISO 3166-1 alpha-2 officiellement assignés (les codes user-assigned ne sont pas reconnus) — la liste est la norme, jamais une invention (F04 §3.2, CLAUDE.md n°2) | `CountryCodeValidator` (`Domain/Identity`) ; testé `CountryCodeValidatorTests` |
| INV-VALIDATION-011 | Le SIREN émetteur de référence vient du PROFIL DU TENANT (CFG02, lu via `TenantSettings.Contracts` scopé par `CompanyId`), jamais du document ; absence/invalidité du profil et incohérence document↔profil sont BLOQUANTES. Le SIRET émetteur porté par le document est validé s'il est fourni (F04 §3.1, 14 chiffres + Luhn) → anomalie BLOQUANTE. (F04 §3.1, note VAL02, CLAUDE.md n°3/9) | `SupplierIdentityRule` (`Domain/Rules`) ; testé `SupplierIdentityRuleTests` |
| INV-VALIDATION-012 | L'identité acheteur est contrôlée sans inventer d'obligation : SIREN acheteur validé s'il est présent, code pays acheteur validé s'il est renseigné ; toute incohérence est BLOQUANTE (F04 §3.2, CLAUDE.md n°3) | `BuyerIdentityRule` (`Domain/Rules`) ; testé `BuyerIdentityRuleTests` |
| INV-VALIDATION-013 | Toute ligne exonérée (catégorie E à taux 0) sans code VATEX est BLOQUANTE (F04 §3.4 — l'absence de VATEX = blocage silencieux côté PA) | `VatexRequiredRule` (anomalie `VATEX_MISSING`, sévérité `Blocking`) ; testé `VatexRequiredRuleTests` |
| INV-VALIDATION-014 | La cohérence catégorie/taux est BLOQUANTE sur les 9 catégories UNCL5305 (S/AA/AAA → taux > 0 ; Z/E/AE/G/K/O → taux 0) ; aucune règle fiscale inventée (F03 §2.1, F04 §3.4 amendée) | `CategoryRateConsistencyRule` (anomalie `CATEGORY_RATE_INCONSISTENT`) ; testé `CategoryRateConsistencyRuleTests` |
| INV-VALIDATION-015 | Un régime source non couvert par la table de mapping (catégorie non résolue post-F03) bloque le document — FAIL-SAFE, jamais d'envoi à l'aveugle (F04 §3.4) | `MappingCoverageRule` (anomalie `MAPPING_COVERAGE_MISSING`) ; testé `MappingCoverageRuleTests` |
| INV-VALIDATION-016 | Un avoir (référence d'origine présente) n'est transmissible que si sa facture d'origine est CONNUE et DÉJÀ ÉMISE (sinon orphelin bloqué, jamais de référence fabriquée) ET si tous ses montants sont POSITIFS (la nature « avoir » est portée par le type, pas le signe — F07-F08 §B.2/§B.4) | `CreditNoteRule` (anomalies `CREDIT_NOTE_REF_MISSING` / `CREDIT_NOTE_ORPHAN` / `CREDIT_NOTE_ORIGINAL_NOT_ISSUED` / `CREDIT_NOTE_NEGATIVE_AMOUNT`) via `IIssuedInvoiceLookup` tenant-scopé ; testé `CreditNoteRuleTests` |
| INV-VALIDATION-017 | La réconciliation des totaux est SANS tolérance, en `decimal`, après arrondi half-up 2 décimales (CLAUDE.md n°1 ; EN 16931) : HT (BT-109) = Σ lignes − remises document + charges document (BR-CO-13), TTC = HT + TVA (BR-CO-15) ; un écart même d'un centime est BLOQUANT. La réconciliation TVA des lignes n'est exécutée qu'en l'absence de charge/remise de niveau document (TVA non résolue avant TVA04) | `LineTotalsRule`, `ArithmeticRule` (`PivotRounding.RoundAmount`, comparaison stricte) ; testé `LineTotalsRuleTests`, `ArithmeticRuleTests` |
| INV-VALIDATION-018 | L'écart total passerelle ↔ total source est une ALERTE (Warning), jamais un blocage — niveau de la spec, pas un affaiblissement (F04 §3.3, décision #3/D5) | `SourceTotalsRule` (sévérité `Warning`, comparaison seulement si `SourceTotalGross` présent) ; testé `SourceTotalsRuleTests` |
| INV-VALIDATION-019 | Un document sans ligne, une date d'émission dans le futur, ou une devise hors ISO 4217 sont BLOQUANTS ; une date antérieure à l'an 2000 est une ALERTE (F04 §3.3, décision #4) | `StructureRule` ; devise vérifiée contre `Iso4217Currencies` (référentiel ISO 4217, F04 §2.4) ; testé `StructureRuleTests` |
| INV-VALIDATION-020 | La détection « date dans le futur » s'appuie sur une horloge injectée (`TimeProvider`), jamais sur `DateTime.Now` en dur (déterminisme, testabilité) | `StructureRule` (constructeur `TimeProvider`) ; testé `StructureRuleTests` |
| INV-VALIDATION-021 | Le numéro de document doit être présent ET ne pas avoir déjà été émis pour le tenant ; les deux contrôles sont BLOQUANTS. Numéro absent ⇒ pas d'interrogation du module Documents (rien à rechercher) | `UniquenessRule` ; testé `UniquenessRuleTests` |
| INV-VALIDATION-022 | La recherche d'unicité est TENANT-SCOPÉE (`CompanyId` du contexte, CLAUDE.md n°9) et passe par le port `IIssuedDocumentLookup` (Contracts-only), jamais par un accès direct au Domain d'un autre module | `UniquenessRule` + `IIssuedDocumentLookup` ; testé `UniquenessRuleTests` (`Lookup_is_tenant_scoped`) |
| INV-VALIDATION-023 | Garde-fou B2B/B2C : un acheteur (traité en B2C) qui présente un indice professionnel de F07-F08 §A.4 — indice « société » brut, n° de TVA intracommunautaire présent, ou forme juridique dans la raison sociale — est BLOQUÉ (`BUYER_LOOKS_PROFESSIONAL`). Toute l'heuristique vit sur la plateforme (l'agent ne transmet que des champs source bruts) ; la liste de formes juridiques (`SARL, SAS, SA, EURL, EI`) et les indices sont EXACTEMENT ceux de la spec, aucun ajout ni seuil inventé ; le montant n'est JAMAIS un critère (garanti structurellement) (F08, CLAUDE.md n°2/3/6) | `CompanyHintDetector` (`Domain/Detection`) + `BuyerLooksProfessionalRule` (`Domain/Rules`) ; testé `CompanyHintDetectorTests`, `BuyerLooksProfessionalRuleTests` |
