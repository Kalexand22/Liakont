# Ingestion Module — Invariants

| ID | Rule | Enforcement |
|---|---|---|
| INV-INGESTION-001 | La clé API d'un agent n'est JAMAIS stockée ni renvoyée en clair : seuls le `key_prefix` (public) et l'empreinte SHA-256 (`key_hash`) sont persistés. La clé complète n'existe qu'à l'émission (enregistrement/rotation) et n'est révélée qu'UNE fois | `AgentKey` (génération + hash) ; `Agent.KeyHash` ; `AgentKeyIssuedDto.FullKey` renvoyé seulement par `RegisterAgent`/`RotateAgentKey` ; `AgentSummaryDto` n'expose pas la clé ; `PostgresAgentQueries` ne SELECT pas `key_hash` |
| INV-INGESTION-002 | Le registre d'agents vit dans la base SYSTÈME (résolution clé → tenant avant tout contexte tenant). Chaque ligne porte son `tenant_id` ; toute opération de GESTION (register/revoke/rotate/list) est scopée au tenant courant | `ISystemConnectionFactory` ; `IngestionTenantScope.Require(ITenantContext)` dans chaque handler de gestion ; `GetByIdForTenantAsync` + clause `WHERE tenant_id` |
| INV-INGESTION-003 | Le `key_prefix` est unique sur toute l'instance (résolution déterministe d'une clé) | Index unique `uq_agents_key_prefix` ; collision → `ConflictException` (`PostgresAgentRegistryUnitOfWork.InsertAsync`) |
| INV-INGESTION-004 | Une clé absente/mal formée/inconnue ou d'empreinte non concordante → 401 ; une clé connue mais révoquée → 403. Au niveau de l'issue, « préfixe inconnu » et « empreinte non concordante » sont indistinguables (anti-énumération) ; le coût d'empreinte est égalisé entre ces deux chemins | `AgentAuthenticator` (vérifie l'empreinte AVANT l'état de révocation, comparaison en temps constant `AgentKey.HashesMatch` ; le chemin « préfixe inconnu » appelle `Agent.SpendKeyComparisonTime` pour égaliser le temps) ; `AgentApiAuthenticationFilter` (mappe les issues vers 401/403) |
| INV-INGESTION-005 | Une version de contrat absente, inconnue ou trop ancienne (< N-1) → 426 Upgrade Required ; la plateforme supporte N et N-1 | `AgentContractVersionPolicy.IsSupported` ; `AgentApiAuthenticationFilter` (426 avant l'authentification) |
| INV-INGESTION-006 | L'historique des heartbeats est append-only : aucun chemin d'update/delete applicatif. La rétention 90 j relève de la supervision (pas une piste d'audit légale) et est assurée hors de ce module | `IAgentRegistryUnitOfWork` n'expose que `AppendHeartbeatAsync` (insert) ; aucune méthode d'update/delete sur `agent_heartbeats` |
| INV-INGESTION-007 | La configuration renvoyée à l'agent est un DÉFAUT SÛR tant que ses sources n'existent pas : `updateRequired = false` et champs d'update `null` (registre de versions OPS07 vide) ; `extractionSchedule = null` (repli de l'agent sur sa planification locale). Aucune règle inventée | `SafeDefaultAgentConfigurationProvider` (renvoie `new AgentConfigurationDto()`) — point d'extension unique pour OPS07 et la planification pilotée |
| INV-INGESTION-008 | Un agent révoqué ne peut pas faire pivoter sa clé (on réenregistre un nouvel agent) | `Agent.RotateKey` → `ConflictException` si `IsRevoked` |
| INV-INGESTION-009 | Anti-doublon par tenant : une empreinte de payload (canonique, PIV02) déjà reçue pour le tenant → `duplicate`, AUCUN effet (ni document, ni événement). Évalué AVANT l'altération (un re-push complet est un doublon, pas une altération) | `DocumentIngestionDecision.Evaluate` (doublon prioritaire) ; index unique `uq_received_documents_tenant_payload` ; `ConflictException` sur course concurrente → `duplicate` |
| INV-INGESTION-010 | Détection d'altération (F06) : même `source_reference` pour le tenant avec une empreinte DIFFÉRENTE → `accepted` + événement `SourceAlterationDetected` (consommé par TRK03), en plus de `DocumentReceived` | `DocumentIngestionDecision` (`AcceptedAltered`, `PreviousPayloadHash`) ; `IngestDocumentBatchHandler` publie les deux événements |
| INV-INGESTION-011 | Le registre de réception est append-only : un document reçu est un FAIT, aucun chemin d'update/delete applicatif | `IReceivedDocumentUnitOfWork` n'expose que des lectures + `InsertReceivedDocumentAsync` ; `ReceivedDocument` sans mutateur |
| INV-INGESTION-012 | Toute écriture/lecture de réception (anti-doublon, régimes source) est scopée au `tenant_id` de l'agent AUTHENTIFIÉ, jamais au corps de la requête : un agent n'écrit que dans SON tenant (CLAUDE.md n°9) | `IngestDocumentBatchCommand.TenantId`/`AgentId` posés par l'endpoint depuis l'identité ; clauses `WHERE tenant_id` ; tables portent `tenant_id` |
| INV-INGESTION-013 | Lot NON transactionnel, résultat INDIVIDUEL par document ; un document malformé (référence/numéro manquant, payload non sérialisable) est REJETÉ entièrement (jamais d'acceptation partielle), sans bloquer le reste du lot ; lot > 100 → 413 | `IngestDocumentBatchHandler.ProcessDocumentAsync` (une transaction par document) ; endpoint `MaxBatchDocuments` → 413 ; corps non conforme → 400 |
| INV-INGESTION-014 | PDF stockés par tenant, sans dépendance au module Document du socle ; nom de tenant et de fichier assainis (anti path-traversal), pool conservé distinctement, rattaché adressable par empreinte de `source_reference` (ADR-0008) | `IIngestedPdfStore` / `FileSystemIngestedPdfStore` (`SafeTenant`, `SafeFileName`, `HashToFileName`, `FileMode.CreateNew` pour le pool) |
| INV-INGESTION-015 | Régimes de TVA source persistés BRUTS par tenant, upsert IDEMPOTENT (occurrences = dernière observation, remplacée et non cumulée → un retry ne double-compte pas) ; persistance best-effort, son échec ne casse pas l'ingestion des documents ; aucune interprétation fiscale (CLAUDE.md n°2) | `ISourceTaxRegimeWriter` / `PostgresSourceTaxRegimeWriter` (`ON CONFLICT ... DO UPDATE SET occurrences = EXCLUDED.occurrences`) ; `IngestDocumentBatchHandler.PersistSourceTaxRegimesBestEffortAsync` (try/catch + log, après la boucle documents) |
| INV-INGESTION-016 | LIMITE V1 (détection d'altération sous concurrence) : la décision d'altération lit le dernier hash d'une `source_reference` AVANT l'insertion. Pour une référence INÉDITE, deux pushs strictement CONCURRENTS portant des payloads différents peuvent tous deux être classés « nouveau » (chacun lit `existingHash = null`) — l'index unique `(tenant_id, payload_hash)` n'entre pas en conflit (empreintes distinctes) et aucun `SourceAlterationDetected` n'est émis pour le second. Hypothèse de conception : un agent draine en SÉRIE (F12 §3.1, push séquentiel), donc deux réceptions simultanées de la même référence neuve sont pathologiques. L'anti-doublon par empreinte reste correct en toutes circonstances ; seule la détection d'altération de ce cas-limite est best-effort. À durcir (ré-évaluation post-insert) si un débit concurrent par référence devient réel | `DocumentIngestionDecision.Evaluate` (lecture pré-insert) ; sérialité garantie côté agent (F12 §3.1) |
| INV-INGESTION-017 | STAGING DURABLE À L'INTAKE (PIP00, ADR-0014) — invariant d'ORDRE, pas d'atomicité : pour tout document ACCEPTÉ, le pivot COMPLET est stagé (écrit + contenu flushé sur disque) AVANT le commit du registre `received_documents` + de l'événement outbox `DocumentReceivedV1` ; le pipeline relit le pivot depuis le staging. Un échec de staging avant le commit annule la transaction → au pire un blob ORPHELIN (purgeable, adressé par contenu donc ré-écrit idempotemment au renvoi de l'agent), jamais un événement sans contenu. NUANCE de durabilité : le renommage atomique qui publie le blob au chemin canonique n'est pas fsyncé (pas d'API portable .NET) ; sous coupure d'alimentation entre ce renommage et le commit Postgres, l'événement peut survivre sans le blob — ce cas (PAS une perte) est rattrapé par le FILET DE SÉCURITÉ de l'agent (ADR-0014), qui re-pousse jusqu'au statut `Processed`. Le contenu n'est PLUS jeté à l'intake. Un DOUBLON DÉJÀ RANGÉ ne re-stage pas (terminal) ; un doublon REÇU MAIS NON RANGÉ re-stage et re-tente le rangement (ADR-0012, voir INV-INGESTION-018) | `IngestDocumentBatchHandler.ProcessDocumentAsync` (`CanonicalJson.Serialize` une fois ; `_stagingStore.WriteAsync(...)` AVANT `uow.CommitAsync`) ; `IPayloadStagingStore` (Staging, ADR-0014) — testé `Ingestion.Tests.Integration` |
| INV-INGESTION-018 | AFFINAGE DÉDOUBLONNAGE (ADR-0012) : sur un doublon par empreinte, l'ingestion DISTINGUE « déjà rangé » (le `Detected` existe → terminal, aucun effet) de « reçu mais NON rangé » (le `Detected` n'existe pas → re-stage + re-tente le rangement, idempotent sur `DocumentId` avec l'identité de la réception d'origine). Ferme la perte silencieuse d'un document reçu mais jamais entré dans le pipeline. Le registre `received_documents` reste APPEND-ONLY (aucune ré-inscription) ; l'intake n'est PAS rendu bloquant (risque d'orphelin évité, TRK02) | `IngestDocumentBatchHandler.RetryRangingIfNotRangedAsync` (`IDocumentIntake.IsDocumentRangedAsync` + `GetDocumentIdByPayloadHashAsync` + re-`WriteAsync` + `RegisterDetectedDocumentAsync`) — testé `DocumentBatchIngestionIntegrationTests` (intake échoué → renvoi = rangé ; déjà rangé → pas de re-rangement) |
