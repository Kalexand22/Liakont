# Invariants — module Payments

| ID | Rule | Enforcement |
|---|---|---|
| INV-PAYMENTS-001 | Les montants (`amount`, `taxable_base`, `vat_amount`) sont en `decimal` exact, jamais flottant (CLAUDE.md n°1). Round-trip base ↔ decimal sans perte. Aucun montant à plus de 2 décimales n'est accepté (la colonne `numeric(18,2)` le tronquerait silencieusement — intégrité de stockage, CLAUDE.md n°4). Le taux `vat_rate` est en `numeric(6,4)` (rejet au-delà de 4 décimales). | Colonnes `numeric(18,2)` / `numeric(6,4)` (`V002`/`V003`) ; propriétés `decimal` ; gardes `MonetaryScale.RequireAmount` / `RequireRate` dans `Payment.Create` et `PaymentAggregate.Create` ; tests `Create_Rejects_Amount_With_More_Than_2_Decimals`, `Create_Rejects_Vat_Rate_With_More_Than_4_Decimals`, `Payment_Round_Trips_Decimal_Amount_Without_Loss`, `Aggregate_Round_Trips_Amounts_And_Rate_Without_Loss`. |
| INV-PAYMENTS-002 | Le module ne calcule ni n'interprète rien : il PERSISTE l'encaissement brut tel que reçu de l'agent et l'agrégat tel que CALCULÉ par le pipeline (PIP03). V1 = Flux 10.4 domestique uniquement (F09 amendement D2). | `Payment.Create` / `PaymentAggregate.Create` reportent les valeurs sans calcul ; `PivotPaymentMapper.ToPayment` projette sans interprétation ; tests `PivotPaymentMapperTests`, `Create_Keeps_Source_Amount_And_Fields`. |
| INV-PAYMENTS-003 | Les montants peuvent être NÉGATIFS (trop-perçu / remboursement via rectificatif — F09 §5.4) : aucun garde-fou de positivité. | Aucune contrainte de signe (`Payment`, `PaymentAggregate`, colonnes `numeric`) ; tests `Create_Allows_Negative_Amount_For_Refund`, `Create_Allows_Negative_Amounts_For_Rectification`. |
| INV-PAYMENTS-004 | La création (paiement et agrégat) est IDEMPOTENTE sur l'identifiant : un re-push n'insère ni doublon ni événement de genèse dupliqué, et n'écrase pas un état déjà avancé. | `INSERT … ON CONFLICT (id) DO NOTHING` (`PostgresPaymentUnitOfWork.SavePaymentAsync` / `CreateAggregateAsync`, événement de genèse écrit seulement si insertion) ; tests `SavePayment_Is_Idempotent_On_Id`, `CreateAggregate_Is_Idempotent_On_Id`. |
| INV-PAYMENTS-005 | La piste d'audit `payment_aggregate_events` est APPEND-ONLY (CLAUDE.md n°4) : aucun chemin d'update/delete applicatif, et des triggers base REJETTENT tout UPDATE/DELETE d'une entrée ET tout TRUNCATE de la table — garantie indépendante du code. Un agrégat transmis à la DGFiP a la même valeur fiscale qu'un document. | `IPaymentUnitOfWork` n'expose que `AppendAggregateEventAsync` (aucun update/delete) ; triggers `trg_payment_aggregate_events_append_only` + `trg_payment_aggregate_events_no_truncate` (`V004`) ; tests `Update_Of_An_Event_Is_Rejected`, `Delete_Of_An_Event_Is_Rejected`, `Truncate_Of_The_Audit_Table_Is_Rejected`. |
| INV-PAYMENTS-006 | Aucune purge automatique des tables d'audit (rétention 10 ans, art. L.123-22 — F06 §6 / F09) : aucun code de delete/purge dans le produit V1. | Aucune méthode de suppression dans le repository/UoW ; triggers append-only sur `payment_aggregate_events` ; pas de migration de purge. |
| INV-PAYMENTS-007 | La machine à états de transmission de l'agrégat est EXPLICITE et FERMÉE (TRK04) : seules les transitions autorisées sont permises ; toute autre est refusée par exception AVANT toute mutation (un état refusé n'altère jamais l'agrégat — CLAUDE.md n°3). `Transmitted` et `RejectedByPa` n'ont aucune transition sortante. Chaque transition PRODUIT son `PaymentAggregateEvent`, persisté ATOMIQUEMENT avec l'état. Une transmission (acceptée/rejetée) archive ses snapshots de preuve (payload + réponse PA). | `PaymentAggregateStateMachine.AllowedTransitions` + `EnsureCanTransition` ; `PaymentAggregate.ApplyTransition` ; read-modify-write `IPaymentUnitOfWork.GetAggregateForUpdateAsync` (`SELECT … FOR UPDATE`) + `UpsertAggregateAsync` + `AppendAggregateEventAsync` ; value object `AggregateTransmissionSnapshots` ; tests `PaymentAggregateTests`, `Transmission_Persists_State_And_Proof_Snapshots_As_Jsonb`, `Transition_Is_Not_Visible_Until_Commit`, `Illegal_Transition_Leaves_Aggregate_And_Audit_Untouched`. |
| INV-PAYMENTS-008 | Isolation tenant PAR LA CONNEXION (database-per-tenant, blueprint §7) : le schéma vit dans la base DU TENANT, aucune colonne de tenant, aucune requête cross-tenant possible (CLAUDE.md n°9/17). | Lectures via `IConnectionFactory` (tenant courant) ; UoW sur `TransactionScope` (connexion scopée) ; schéma `payments` sans colonne tenant. |
