# REDLINE — F19 « GED dynamique & coffre-fort documentaire »

> **Statut :** revue critique adversariale de `docs/conception/F19-GED-Dynamique-Coffre-Fort.md` (proposition NON RATIFIÉE).
> **Date :** _TODO (à dater à la diffusion — non inventée)._
> **Relecteur :** revue indépendante, sceptique, vérifiée contre le code réel (`src/Modules/**`, `src/Common/**`, `src/Contracts/**`, `orchestration/**`, `docs/**`).
> **Méthode :** chaque affirmation vérifiable de F19 a été contrôlée contre le code (`file:line` cités). 42 findings adjugés.

## Verdict global

**F19 n'est PAS ratifiable en l'état.** Le document est solide sur la généricité (méta-modèle non-EAV, garde anti-vocabulaire métier, frontières NetArchTest), mais **cinq P1 structurants doivent être corrigés AVANT toute ratification**, car ils invalident des promesses centrales de la spec ou décrivent une mécanique qui, implémentée telle quelle, produirait une perte/corruption de données :

- **RL-01** — contradiction interne : réutilisation d'un type d'`Ingestion.Domain` (`DocumentIngestionDecision`) que la propre table de frontières Contracts-only de F19 interdit.
- **RL-02** — invariant mono-valeur sans garde de concurrence → double « valeur courante » figée et **non réparable** sous append-only.
- **RL-03** — registre d'ingestion GED en base TENANT mais atomique avec l'outbox (base SYSTÈME) : atomicité impossible → **perte silencieuse d'indexation**.
- **RL-05** — promesse « purement additive, hash inchangé, zéro surcoût » **fausse** : la chaîne `documents.archive_entries` exige un `document_id NOT NULL` + FK qu'un document GED-seul n'a pas.
- **RL-07** — numéro de migration `V0NN` non alloué sur un schéma `documents` **déjà à double-V010**, + FK forward-reference contre l'ordre alphabétique DbUp.

Aucun finding n'a été réfuté. Un finding stratégique (RL-08, ajusté P2) recommande en outre de **différer le seeding du chantier GED** tant qu'aucun client e-invoicing n'est en production réelle (voir §7).

## 3. Findings P1 (bloquants — à corriger AVANT ratification)

Consommer un type d'`Ingestion.Domain` depuis `Ged` est une violation de frontière que `module-rules.md` et F19 elle-même classent P1 (`CLAUDE.md` n°14). ### RL-02 | [Confirmé] | §§3.4.3 (NB mono-valeur) + §3.7 (`AppendAxisLinkAsync`) + §8

**Problème.** L'unicité « une seule valeur courante par (document, axe mono) » repose UNIQUEMENT sur le Domain (§3.4.3, l.286 : « portée côté Domain à l'écriture », « une contrainte EXCLUDE n'est pas applicable »), sans contrainte/verrou DB. Sous READ COMMITTED (défaut du projet — `src/Common/Infrastructure/Database/TransactionScope.cs:26` appelle `BeginTransactionAsync(ct)` sans `IsolationLevel`), deux `AppendAxisLinkAsync` concurrents lisent chacun « courante=X », insèrent Y1(supersedes=X) et Y2(supersedes=X), ne se bloquent pas (lignes distinctes, aucun index unique sur `supersedes_id` ni sur la courante) → `current_axis_links` renvoie **DEUX** lignes pour un axe mono. **Pire que le précédent fiscal** (`INV-INGESTION-016`) : aucun index ne rattrape la course, et les triggers append-only §3.6 interdisent UPDATE/DELETE → l'état double-courante est **PERMANENT et non réparable** (violation durable de la règle 4). Le test d'acceptance §8 (l.748) et GED03 (l.831) sont **séquentiels** → passent vert en masquant la course (faux-vert).

**Correction F19.** Matérialiser un mécanisme anti-race et le couvrir par un test **concurrent** : dans `AppendAxisLinkAsync`, avant de superséder un axe MONO, acquérir `pg_advisory_xact_lock(hashtext(managed_document_id::text || ':' || axis_id::text))` OU `SELECT ... FOR UPDATE` sur la ligne `managed_documents` parente, dans la même transaction. Documenter le niveau d'isolation requis. **Ne PAS** se contenter de « documenter comme limite assumée » (le mode de défaillance est une corruption permanente). Corriger §3.4.3:286 et §8:748 + ajouter une acceptance concurrente à GED03.

### RL-03 | [Confirmé] | §§2.4 + §3.2(a) + §4.3.1 (`ged_received_documents`)

**Problème.** §3.2(a) range `ged_received_documents` en base TENANT (« l'isolation EST la connexion »), tout en lui ajoutant `tenant_id` « par symétrie ». Mais §2.4 (l.78-81) exige que `ManagedDocumentReceivedV1` soit écrit dans l'outbox DANS LA MÊME TRANSACTION que l'insert du registre (« déclencheur DURABLE »). Or l'outbox vit en base SYSTÈME, et le seul draineur `OutboxWorker` lit/écrit uniquement la base système via `ISystemConnectionFactory` (`OutboxWorker.cs:70,77,156`). Le code fiscal résout exactement ce point en **co-localisant registre+outbox en base système** : `PostgresReceivedDocumentUnitOfWork.cs:13-18` (« ouverte sur la base SYSTÈME … atomiques ») ; `V004__create_received_documents_table.sql:1-2` (« Vit dans la base SYSTÈME »). **C'est pourquoi `received_documents` porte `tenant_id`.** Il n'existe pas de 2PC entre deux bases PG :
- outbox en base tenant → l'événement n'est **jamais drainé** (faux-vert : l'INSERT commit, l'événement n'est jamais livré → document GED ingéré mais jamais indexé = **perte silencieuse**) ;
- outbox en base système + registre en base tenant → « même transaction » **impossible**.

**Correction F19.** Trancher en faveur de la **base SYSTÈME** : déplacer `ged_received_documents` ET l'écriture de `ManagedDocumentReceivedV1` en base système (schéma dédié `ged_ingestion`), drainé par l'`OutboxWorker` système, via une UoW calquée sur `PostgresReceivedDocumentUnitOfWork`. Corriger §3.2(a) et §4.3.1 (exception db-per-tenant en base système, comme `ingestion.received_documents`). Préciser en §2.4 que la « UNE transaction » est ouverte sur la base système ; l'indexation GED (`managed_documents`, liens, base tenant) est peuplée par le **consommateur asynchrone**. Test anti-régression : `ManagedDocumentReceivedV1` est effectivement drainé et le consommateur crée le `ManagedDocument` ; rien n'écrit dans un outbox de base tenant.

### RL-05 | [Confirmé] | §§1.3 + §5.1 + §3.1 / §4.1 + §5.4

**Problème.** L'argument central de F19 (§1.3 « zéro surcoût, zéro surface ajoutée » ; §5.1 « RÉUTILISE inchangé … la chaîne `documents.archive_entries`, `HashChain`, `ArchiveVerifier` », l.510 ; INV-ARCH-GED-1 « rien à changer », l.539) est **faux** dès qu'on scelle du non-fiscal. Vérifié : `documents.archive_entries.document_id` est `uuid NOT NULL` + FK vers `documents.documents` (`V005__create_archive_entries_table.sql:11,18-20`), et tout le chemin d'écriture l'exige (`IArchiveEntryStore.ReserveAsync(Guid documentId, …)` non-nullable, `Application/IArchiveEntryStore.cs:29` ; `PostgresArchiveEntryStore.InsertSql` insère toujours `document_id`, `Infrastructure/PostgresArchiveEntryStore.cs:45-50` ; `ArchiveEntryRecord.DocumentId` `Guid` non-nullable). La chaîne est **unique par tenant** (`HeadSql` lit la tête globale ; `ArchiveVerifier.cs:48` la lit via `_entryStore.GetChainAsync()`). Or un document GED purement métier n'a **aucune** ligne `documents.documents` (§3.1 l.112, §4.1 l.394). Sceller un document GED-seul force soit une chaîne distincte (que §5.1 interdit), soit `document_id` nullable + allègement de FK = **modification du cœur fiscal à échéance légale**. Aggravant : §5.4 `document_storage_refs.archive_entry_id` FK vers `documents.archive_entries(id)` (l.588) présuppose déjà la ligne `archive_entries`.

**Correction F19.** Reformuler §1.3/§5.1 : la GED **n'est PAS** additive hash-neutre sur la chaîne fiscale quand elle scelle du non-fiscal. Trancher AVANT tout seed entre : **(A)** chaîne d'archive GED **distincte** (table/tête/verifieur propres — alors §5.1 cesse de prétendre « réutilise inchangé ») ; **(B)** `archive_entries.document_id` nullable SANS la FK fiscale pour les `entryKind` GED (modifie V005, `ReserveAsync` en `Guid?`, `InsertSql`, `ArchiveEntryRecord`, et chaîne MIXTE où un paquet GED corrompu romprait la vérification fiscale — à chiffrer comme **impact fiscal**). Retirer « zéro surcoût / purement additive » du cas GED-seul ; limiter la preuve golden §8 au seul paquet FACTURE. **Si non tranché : bloquer GED07** (CLAUDE.md n°2/3).

### RL-07 | [Confirmé] | §§5.4 (`documents.V0NN`) + §3.3.1 (FK `axis_definitions→entity_types`) + §10 (GED03)

**Problème.** **(1)** §5.4 ajoute `documents.V0NN__create_document_storage_refs_table.sql` avec un numéro **non résolu**. Le schéma `documents` porte DÉJÀ deux migrations V010 distinctes (`V010__add_mandant_id_and_unique_index_389.sql` ET `V010__add_pa_transmission_journal_to_document_events.sql` — précédent de collision vérifié) et est touché par plusieurs lots parallèles : un `V0NN` non alloué dans une spec ratifiée = collision/ordre indéterministe garanti au seed. **(2)** §3.3.1 définit `axis_definitions` avec une FK vers `entity_types` (l.143-144) défini **plus bas** (§3.3.2, l.157). **Correction F19.** Allouer un numéro de migration **concret réservé** pour `document_storage_refs` (verrouillé dans l'item, pas « V0NN »). Pour la FK : créer `entity_types` AVANT `axis_definitions` (même fichier, ou nommage garantissant l'ordre alpha), OU poser la FK en `ALTER TABLE` séparé après les deux CREATE. Ajouter une acceptance GED03 « migrations s'appliquent sur base vierge dans l'ordre DbUp, FK satisfaite ».

## 4. Findings P2 (importants, non bloquants pour la ratification — à corriger avant seeding du lot GED)

### RL-04 | [Ajusté] | §§2.4 + §3.4.1 — Double source de création du `ManagedDocument` sans clé d'idempotence

### RL-06 | [Ajusté] | §§11 (D3) + §5.1 — Crypto-shredding RGPD présenté en reco mais non implémentable

La tension WORM-10-ans vs droit à l'effacement est **nommée, pas résolue**. **Fix :** ne pas présenter le crypto-shredding comme reco implémentable ; soit poser explicitement une couche de chiffrement-par-document (ADR sécurité, owner D9), soit acter D3 comme tension ouverte sans mécanisme V1.
